L2 Reference Manual Overview

A draft of the design document is here

The Livingstone2 system is comprised of the libraries described below.   Each layer is dependent upon the layer below, but need not be used with the layers above. 

NOTE: If you are viewing this on your own machine and the links below do not work, you need to build the documentation with Doxygen. To download Doxygen, follow the link at the bottom this page.

Livingstone

The Livingstone library provides discrete state estimation (diagnosis) and reconfiguration (recovery). 

 

The library is able to read in a model of a system to be tracked.  An API allows the user to present commands and observations to Livingstone and receive state estimates in response.   Livingstone uses the commands issued to the system and the observations that were generated as a result in order to determine and track the likely trajectories of the system.

 

Reconfiguration is not yet implemented.

CBFS Search

This library implements a conflict-directed, best-first search that Livingstone can employ in order to choose the likely trajectories to track.   It performs a search over the Transition System representation.

Conflict Coverage Search

This library implements a conflict-coverage that Livingstone can employ in order to choose the likely trajectories to track.   It performs a search over the Transition System representation.

Transition System

The Transition System represents the discrete state variables of the system being tracked and their evolution over time. The Transition System represents recent history exactly and past history approximately.  It is a specialization of the Conflict DB, which is not temporal.  

Conflict DB

This library provides a database of discrete-valued variables and constraints between variable assignments.    It provides the notion of a conflict, or a set of assignments that cannot all be made at once, as they violate the constraints.

Truth Maintenance System

The TMS accepts a theory consisting of set of propositions and constraints between them.  Given a truth assignment to a subset of the propositions, it determines if the theory is currently consistent.  If not, it returns a conflict between assignments that is causing an inconsistency.  This is a standard LTMS with some extensions to allow it to support the creation and elimination of propositions and constraints over time.

MBA Utilities

This library contains a set of basic utility classes such as Lists and Hash Tables.  The interfaces conform to the C++ Standard Template Library except where noted. 

 

These descriptions need to be worked and generalized beyond from what the current implementation does to the basic functionality provided by each layer.  For example, the TMS does not have to be propositional.
